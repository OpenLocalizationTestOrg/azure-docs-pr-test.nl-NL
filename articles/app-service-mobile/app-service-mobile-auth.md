## <a name="how-authorization-works"></a><span data-ttu-id="f27ce-145">De werking van autorisatie</span><span class="sxs-lookup"><span data-stu-id="f27ce-145">How authorization works</span></span>

<span data-ttu-id="f27ce-148">Een mogelijkheid is het niet-geverifieerde aanvragen omleiden naar een van de id-providers hebben.</span><span class="sxs-lookup"><span data-stu-id="f27ce-148">One option is to have unauthenticated requests redirect to one of the identity providers.</span></span> <span data-ttu-id="f27ce-149">In een webbrowser, zou dit daadwerkelijk de gebruiker naar een nieuwe pagina duren.</span><span class="sxs-lookup"><span data-stu-id="f27ce-149">In a web browser, this would actually take the user to a new page.</span></span> <span data-ttu-id="f27ce-150">Echter uw mobiele clients niet op deze manier worden omgeleid en niet-geverifieerde antwoorden ontvangt een HTTP *401 niet geautoriseerd* antwoord.</span><span class="sxs-lookup"><span data-stu-id="f27ce-150">However, your mobile client cannot be redirected in this way, and unauthenticated responses will receive an HTTP *401 Unauthorized* response.</span></span> <span data-ttu-id="f27ce-151">Daardoor vormen de eerste aanvraag de client maakt moet altijd het eindpunt van de aanmelding en vervolgens kunt u aanroepen naar andere API's.</span><span class="sxs-lookup"><span data-stu-id="f27ce-151">Given this, the first request your client makes should always be to the login endpoint, and then you can make calls to any other APIs.</span></span> <span data-ttu-id="f27ce-152">Als u een ander API aanroepen probeert voordat u zich aanmeldt, ontvangt de client een fout opgetreden.</span><span class="sxs-lookup"><span data-stu-id="f27ce-152">If you attempt to call another API before logging in, your client will receive an error.</span></span>

<span data-ttu-id="f27ce-153">Als u meer gedetailleerde wilt laten bepalen via welke eindpunten verificatie vereisen, kunt u kiezen "Er is geen actie (toestaan aanvraag) ' voor niet-geverifieerde aanvragen.</span><span class="sxs-lookup"><span data-stu-id="f27ce-153">If you wish to have more granular control over which endpoints require authentication, you can also pick "No action (allow request)" for unauthenticated requests.</span></span> <span data-ttu-id="f27ce-154">In dit geval worden alle beslissingen voor de verificatie uitgesteld de toepassingscode.</span><span class="sxs-lookup"><span data-stu-id="f27ce-154">In this case, all authentication decisions are deferred to your application code.</span></span> <span data-ttu-id="f27ce-155">Hierdoor kunt u ook toegang tot specifieke gebruikers op basis van aangepaste autorisatieregels.</span><span class="sxs-lookup"><span data-stu-id="f27ce-155">This also allows you to allow access to specific users based on custom authorization rules.</span></span>

## <a name="partitions-in-azure-storage"></a><span data-ttu-id="e715d-125">Partities in Azure-opslag</span><span class="sxs-lookup"><span data-stu-id="e715d-125">Partitions in Azure Storage</span></span>
<span data-ttu-id="e715d-126">Elk object dat bevat gegevens die zijn opgeslagen in Azure Storage (BLOB's, berichten, entiteiten en bestanden) behoort tot een partitie en wordt geïdentificeerd door een partitiesleutel.</span><span class="sxs-lookup"><span data-stu-id="e715d-126">Every object that holds data that is stored in Azure Storage (blobs, messages, entities, and files) belongs to a partition, and is identified by a partition key.</span></span> <span data-ttu-id="e715d-127">De partitie bepaalt hoe Azure Storage load compromis tussen de BLOB's, berichten, entiteiten en bestanden op servers om te voldoen aan de behoeften van verkeer van die objecten.</span><span class="sxs-lookup"><span data-stu-id="e715d-127">The partition determines how Azure Storage load balances blobs, messages, entities, and files across servers to meet the traffic needs of those objects.</span></span> <span data-ttu-id="e715d-128">De partitiesleutel is uniek en wordt gebruikt om te zoeken naar een blob, het bericht of de entiteit.</span><span class="sxs-lookup"><span data-stu-id="e715d-128">The partition key is unique and is used to locate a blob, message, or entity.</span></span>

<span data-ttu-id="e715d-129">Bovenstaande tabel [Schaalbaarheidsdoelen voor Standard-Opslagaccounts](#standard-storage-accounts) geeft een lijst van de prestatiedoelen voor één partitie voor elke service.</span><span class="sxs-lookup"><span data-stu-id="e715d-129">The table shown above in [Scalability Targets for Standard Storage Accounts](#standard-storage-accounts) lists the performance targets for a single partition for each service.</span></span>

<span data-ttu-id="e715d-130">Partities invloed hebben op taakverdeling en schaalbaarheid voor elk van de storage-services in de volgende manieren:</span><span class="sxs-lookup"><span data-stu-id="e715d-130">Partitions affect load balancing and scalability for each of the storage services in the following ways:</span></span>

* <span data-ttu-id="e715d-131">**BLOBs**: de partitiesleutel voor een blob is accountnaam + containernaam + blob-naam.</span><span class="sxs-lookup"><span data-stu-id="e715d-131">**Blobs**: The partition key for a blob is account name + container name + blob name.</span></span> <span data-ttu-id="e715d-132">Dit betekent dat elke blob een eigen partitie hebben kan als de belasting van de blob is vereist.</span><span class="sxs-lookup"><span data-stu-id="e715d-132">This means that each blob can have its own partition if load on the blob demands it.</span></span> <span data-ttu-id="e715d-133">BLOBs kunnen op veel servers worden gedistribueerd om de toegang tot uitbreiden, maar één blob kan alleen worden geleverd door één server.</span><span class="sxs-lookup"><span data-stu-id="e715d-133">Blobs can be distributed across many servers in order to scale out access to them, but a single blob can only be served by a single server.</span></span> <span data-ttu-id="e715d-134">Blobs kunnen logisch worden gegroepeerd in blob-containers, maar er zijn geen partitionering gevolgen van deze groepering.</span><span class="sxs-lookup"><span data-stu-id="e715d-134">While blobs can be logically grouped in blob containers, there are no partitioning implications from this grouping.</span></span>
* <span data-ttu-id="e715d-135">**Bestanden**: de partitiesleutel voor een bestand is de sharenaam van naam + bestand-account.</span><span class="sxs-lookup"><span data-stu-id="e715d-135">**Files**: The partition key for a file is account name + file share name.</span></span> <span data-ttu-id="e715d-136">Dit betekent dat alle bestanden in een bestandsshare bevinden zich ook in één partitie.</span><span class="sxs-lookup"><span data-stu-id="e715d-136">This means all files in a file share are also in a single partition.</span></span>
* <span data-ttu-id="e715d-137">**Berichten**: de partitiesleutel voor een bericht is de accountnaam + wachtrijnaam, dus alle berichten in een wachtrij zijn ondergebracht in één partitie en door één server worden behandeld.</span><span class="sxs-lookup"><span data-stu-id="e715d-137">**Messages**: The partition key for a message is the account name + queue name, so all messages in a queue are grouped into a single partition and are served by a single server.</span></span> <span data-ttu-id="e715d-138">Verschillende wachtrijen kunnen worden verwerkt door andere servers voor de taakverdeling voor echter veel wachtrijen een opslagaccount kan hebben.</span><span class="sxs-lookup"><span data-stu-id="e715d-138">Different queues may be processed by different servers to balance the load for however many queues a storage account may have.</span></span>
* <span data-ttu-id="e715d-139">**Entiteiten**: de partitiesleutel voor een entiteit is het account naam + tabelnaam + partitie-sleutel, waarbij de partitiesleutel de waarde van de vereiste gebruiker gedefinieerde is **PartitionKey** eigenschap voor de entiteit.</span><span class="sxs-lookup"><span data-stu-id="e715d-139">**Entities**: The partition key for an entity is account name + table name + partition key, where the partition key is the value of the required user-defined **PartitionKey** property for the entity.</span></span> <span data-ttu-id="e715d-140">Alle entiteiten met dezelfde partitie sleutelwaarde in dezelfde partitie worden gegroepeerd en door de server met dezelfde partitie worden behandeld.</span><span class="sxs-lookup"><span data-stu-id="e715d-140">All entities with the same partition key value are grouped into the same partition and are served by the same partition server.</span></span> <span data-ttu-id="e715d-141">Dit is een belangrijk punt om te begrijpen bij het ontwerpen van uw toepassing.</span><span class="sxs-lookup"><span data-stu-id="e715d-141">This is an important point to understand in designing your application.</span></span> <span data-ttu-id="e715d-142">De voordelen van de schaalbaarheid van entiteiten verspreid over meerdere partities met de voordelen van toegang voor het groeperen van entiteiten in één partitie moet worden verdeeld in uw toepassing.</span><span class="sxs-lookup"><span data-stu-id="e715d-142">Your application should balance the scalability benefits of spreading entities across multiple partitions with the data access advantages of grouping entities in a single partition.</span></span>  

<span data-ttu-id="e715d-143">Een groot voordeel op een verzameling entiteiten in een tabel groeperen in één partitie is dat het is mogelijk atomic batchbewerkingen uitvoert via voor entiteiten in dezelfde partitie omdat een partitie op één server bestaat.</span><span class="sxs-lookup"><span data-stu-id="e715d-143">A key advantage to grouping a set of entities in a table into a single partition is that it's possible to perform atomic batch operations across entities in the same partition, since a partition exists on a single server.</span></span> <span data-ttu-id="e715d-144">Overweeg daarom als u uitvoeren van batchbewerkingen op een groep van entiteiten wilt, om ze te groeperen met dezelfde partitiesleutel.</span><span class="sxs-lookup"><span data-stu-id="e715d-144">Therefore, if you wish to perform batch operations on a group of entities, consider grouping them with the same partition key.</span></span> 

<span data-ttu-id="e715d-145">Aan de andere kant entiteiten die zich in dezelfde tabel, maar de verschillende partitiesleutels kan worden met gelijke taakverdeling over verschillende servers, waardoor het mogelijk om grotere schaalbaarheid.</span><span class="sxs-lookup"><span data-stu-id="e715d-145">On the other hand, entities that are in the same table but have different partition keys can be load balanced across different servers, making it possible to have greater scalability.</span></span>

<span data-ttu-id="e715d-146">Aanbevelingen voor het ontwerpen van partitioneren strategie voor tabellen vindt u gedetailleerde [hier](https://msdn.microsoft.com/library/azure/hh508997.aspx).</span><span class="sxs-lookup"><span data-stu-id="e715d-146">Detailed recommendations for designing partitioning strategy for tables can be found [here](https://msdn.microsoft.com/library/azure/hh508997.aspx).</span></span>
